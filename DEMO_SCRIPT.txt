[1:00 - ARCHITECTURE]
(show assets/architecture_diagram.png)
Let's walk through the architecture of the system.
Our entry point begins with the user's natural language input, which first hits the security checkpoint callback node.
This security node acts as a gateway, routing requests to the SECURITY_EVENT block path if it detects prompt injections, or returning a polite refusal if the query contains video game keywords.
If the input contains personal data, the security node scrubs emails, phones, and credit card numbers in-place before returning control to the runner.
Clean queries are then routed to our main agent, board_game_hub_agent, powered by Gemini.
The agent decides which Model Context Protocol, or MCP, tools to execute: search_board_game and recommend_board_games to query the Kaggle BGG CSV database, or add_owned_games, remove_owned_games, and list_owned_games to update the collection.
For scheduling sessions, it calls the organize_gaming_session tool to allocate games into a flexible number of tables under a four-hour budget while loading session history to prevent game repetition.
Finally, humans are kept in the loop conversationally to resolve name ambiguities and review the generated schedules.
